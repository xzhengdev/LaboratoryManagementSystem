# 后期代码改动实施指南（分阶段执行）

> 目标：为后续“分布式重构 + 资产管理 + SeaweedFS + 小程序日报上传”提供可直接执行的代码改动路线。

---

## 1. 改动总原则

1. 先“数据模型与后端能力”，再“前端/小程序页面”。  
2. 先“可跑通主链路”，再“补性能与可观测性”。  
3. 每一阶段都必须有可验证结果（接口可用、测试通过、数据正确）。  
4. 高并发机制不后补，必须在关键写接口改造时同步接入。  

---

## 2. 分阶段改动计划

## 阶段A：数据库与模型层改造（先做）
### 目标
完成资产管理、日报上报、图片元数据、汇总基础表。

### 主要改动
1. 新增迁移脚本：`lab/backend/migrations/`  
2. 新增/扩展模型：`lab/backend/app/models/entities.py`（或拆 `models/assets.py`）  
3. 新增索引与唯一约束（幂等、状态流转、查询条件）。

### 建议新增表
1. `asset_budget`  
2. `asset_purchase_request`  
3. `asset_budget_ledger`  
4. `asset_item`  
5. `asset_photo`  
6. `lab_daily_report`  
7. `lab_daily_report_photo`  
8. `outbox_event`（可选）  

### 验收
1. 迁移可执行且可回滚。  
2. 表结构满足后续接口字段需求。  

---

## 阶段B：基础设施接入（SeaweedFS + 配置 + 路由）
### 目标
接入校区级 SeaweedFS，对图片上传能力完成基础封装。

### 主要改动
1. 配置：`lab/backend/app/config.py`  
   - 增加 SeaweedFS 开关与各校区 `FILER_URL/COLLECTION`。  
2. 新增扩展/客户端封装：`lab/backend/app/services/storage_service.py`  
   - `upload_file(campus_id, file, biz_type)`  
   - `delete_file(campus_id, file_id)`（可选）  
3. 若需要，补充 `extensions.py` 初始化对象。

### 验收
1. 单独测试上传接口可返回 `file_id/url/hash`。  
2. 同一校区请求进入对应 SeaweedFS 节点。  

---

## 阶段C：后端业务接口改造（核心）
### 目标
跑通资产主链路与日报主链路，具备管理员审核能力。

### 主要改动（资产）
1. 新增 API：`lab/backend/app/api/assets.py`、`asset_requests.py`（或合并）  
2. 新增 Service：`lab/backend/app/services/asset_service.py`  
3. 核心接口：  
   - `POST /api/asset-requests`（申报锁额度）  
   - `PUT /api/asset-requests/{id}/approve`  
   - `PUT /api/asset-requests/{id}/reject`  
   - `POST /api/assets`（入库）  
   - `POST /api/assets/photos/upload`（SeaweedFS上传）

### 主要改动（日报）
1. 新增 API：`lab/backend/app/api/lab_reports.py`  
2. 新增 Service：`lab/backend/app/services/lab_report_service.py`  
3. 核心接口：  
   - `POST /api/lab-reports`  
   - `POST /api/lab-reports/photos/upload`  
   - `GET /api/lab-reports/my`  
   - `GET /api/lab-reports`（管理员）  
   - `PUT /api/lab-reports/{id}/review`

### 并发与一致性（本阶段必须接入）
1. 幂等键：复用现有 `IdempotencyRecord` 机制。  
2. 锁机制：预算池与热点资源使用 Redis 锁 + DB 事务。  
3. 状态机：审批流转仅允许合法状态跳转。  

### 验收
1. 资产流程：申报 -> 审批 -> 入库全链路可跑通。  
2. 日报流程：学生上传 -> 管理员查看/审核可跑通。  
3. 并发提交下预算无超额、无重复扣减。  

---

## 阶段D：前端与小程序页面改造
### 目标
提供学生与管理员可用页面。

### 主要改动（小程序）
1. 新增页面：  
   - `lab/miniprogram/pages/lab-report-submit/*`

   - `lab/miniprogram/pages/lab-report-review/*`（管理员）  
2. 更新 API 封装：`lab/miniprogram/utils/api.js`、`request.js`  
3. 上传流程：`wx.chooseMedia` -> 后端上传接口 -> 提交日报。

### 主要改动（Web 管理端）
1. 新增页面：`lab/frontend/pages/admin-lab-reports/index.vue`  
2. 接入列表、筛选、详情、审核操作。

### 验收
1. 学生端拍照上传与提交成功。  
2. 管理端可按校区/实验室/日期筛选并审核。  

---

## 阶段E：汇总、测试与答辩数据
### 目标
完成跨校区总表和论文可用实验结果。

### 主要改动
1. 汇总服务：  
   - 新增 `summary_service.py` 或扩展 `statistics_service.py`  
   - 异步消费 `outbox_event` 更新总表。  
2. 测试脚本：`lab/benchmark/` 新增资产与日报并发脚本。  
3. 故障测试：单实例下线、消息积压、恢复补偿。  

### 验收
1. 中心总表可生成并与分校区数据对齐。  
2. 形成可入论文的对比数据（QPS、P95/P99、冲突率、错误率）。

2. 每阶段提交前执行：

   - 关键并发用例验证

   - `feat(report): add mini-program daily lab report upload`

## 5. 测试清单（必须覆盖）

1. 功能正确性
   - 资产申报/审批/入库状态流转正确。  
   - 日报提交/审核流程正确。  
2. 并发一致性
   - 同时多请求申报不超额。  
   - 同幂等键重复请求无重复写入。  
3. 存储可用性
   - SeaweedFS 上传、读取、异常重试可用。  
4. 故障恢复
   - 单节点异常时核心接口可继续服务。  
   - 汇总任务恢复后可补齐数据。  

---

## 6. 风险与回滚策略

1. 风险：迁移脚本错误导致数据结构不兼容。  
   - 策略：每次迁移前导出备份，迁移脚本可逆。  
2. 风险：SeaweedFS 接入后上传失败影响主流程。  
   - 策略：开关控制，必要时回退本地上传路径。  
3. 风险：并发控制缺失导致预算错账。  
   - 策略：强制幂等 + 锁 + 事务，未通过并发测试不得上线。  

---

## 7. 最终完成判定

满足以下条件可判定“后期改动完成”：
1. 资产与日报两个新业务链路全部上线可用。  
2. 图片全部走校区 SeaweedFS（或兼容期内可配置切换）。  
3. 跨校区总表可稳定生成。  
4. 并发与故障测试结果可支撑论文与答辩。
