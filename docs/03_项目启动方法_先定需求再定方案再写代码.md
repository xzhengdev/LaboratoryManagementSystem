# 项目启动方法文档：先定需求，再定方案，再写代码

## 1. 文档目的
本文件用于统一项目启动方式，避免“边做边改、反复返工”。
执行原则：**先定需求，再定方案，再写代码**。

---

## 2. 三阶段总流程

### 阶段A：先定需求（What & Why）
目标：把“要做什么、为什么做、做到什么算完成”说清楚。

必须产出：
- 需求说明（功能清单）
- 非功能需求（性能、安全、可用性）
- 业务规则清单
- 需求优先级（P0/P1/P2）
- 验收标准（可测试）

完成标准：
- 每条需求可被验证（不是口号）
- 范围边界明确（做/不做）
- 干系人对需求版本达成一致

禁止事项：
- 需求未冻结就进入编码
- 没有验收标准就排开发任务

---

### 阶段B：再定方案（How）
目标：把“怎么实现最稳妥”定下来。

必须产出：
- 技术架构图（模块边界、调用关系）
- 数据库设计（实体、关系、关键约束）
- API 设计（接口、字段、错误码）
- 关键流程图（如登录、主业务流程、状态流转）
- 风险清单与应对策略

完成标准：
- 每个P0需求都映射到具体模块和接口
- 关键数据结构和状态流转无冲突
- 方案可被开发与测试直接执行

禁止事项：
- 没有接口契约就前后端并行开发
- 没有数据库迁移方案就改表上线

---

### 阶段C：再写代码（Build）
目标：按已确认需求和方案进行实现、测试、交付。

必须产出：
- 可运行代码（按优先级迭代）
- 测试用例与测试结果
- 发布说明与变更记录
- 用户文档/使用说明

完成标准：
- P0功能全部通过验收
- 核心流程测试通过
- 缺陷收敛到可接受范围

禁止事项：
- 未通过测试直接上线
- 临时改需求不走变更流程

---

## 3. 项目开工前检查清单（Checklist）

### 3.1 需求就绪检查
- [ ] 项目目标一句话描述清楚
- [ ] 用户角色与核心场景明确
- [ ] 功能需求列表完整
- [ ] 非功能需求已定义
- [ ] 业务规则已列出
- [ ] 验收标准可执行
- [ ] 需求优先级已排序

### 3.2 方案就绪检查
- [ ] 架构设计已评审
- [ ] 数据库设计已确认
- [ ] API 契约已确认
- [ ] 关键流程图已输出
- [ ] 风险与预案已登记

### 3.3 开发就绪检查
- [ ] 任务拆解到人和时间
- [ ] 分支策略和提交流程明确
- [ ] 测试计划明确
- [ ] 发布与回滚方案明确

---

## 4. 角色分工建议
- 产品/需求负责人：需求收敛、优先级、验收标准。
- 架构/后端负责人：方案设计、数据与接口边界。
- 前端负责人：交互流程、页面实现、接口联调。
- 测试负责人：用例设计、回归策略、验收报告。
- 项目负责人：进度、风险、变更控制。

---

## 5. 推荐时间安排（小中型项目）
- 需求阶段：20%
- 方案阶段：20%
- 开发与测试阶段：60%

示例（4周项目）：
- 第1周：需求分析与冻结
- 第2周：方案设计与评审
- 第3-4周：开发、测试、修复、交付

---

## 6. 变更控制规则
如果编码阶段出现新增需求：
1. 先登记变更项。
2. 评估影响（范围、工期、风险）。
3. 重新确认优先级。
4. 更新需求文档与计划后再开发。

原则：**不禁止变更，但禁止“隐形变更”。**

## 8. 一句话执行准则
**没有需求基线，不做方案；没有方案基线，不写代码。**
