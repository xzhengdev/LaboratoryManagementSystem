## 8. “分布式”和“高并发”在系统中的落点（统一口径）

## 8.1 分布式落点

1. 校区级分库（数据自治）
- 实现方式：按 `campus_id` 路由到对应校区库；本地业务只写本地库。
- 解决问题：避免单库瓶颈与跨校区写入耦合，降低故障影响面。
- 验证指标：单校区库异常时，其它校区业务不受影响；跨校区请求路由准确率 100%。

2. 校区级 SeaweedFS 本地存储（资源自治）
- 实现方式：每个校区独立部署 `SeaweedFS(master+volume+filer)`，图片写入本校区对象存储。
- 解决问题：图片不再依赖微信/人工转发，满足“本地存储、校区自治”要求。
- 验证指标：上传成功率、上传耗时、图片元数据入库一致性、跨校区不搬原图。

3. 中心异步汇总总表（协同治理）
- 实现方式：各校区业务变更写事件，中心消费后幂等更新 `summary_fact` 总表。
- 解决问题：避免跨库实时联查；支持全局监管统计。
- 验证指标：汇总延迟（如分钟级）、补偿成功率、总表与分校区源数据一致率。

4. 校区自治 + 全局监管权限模型（组织分布式）
- 实现方式：`lab_admin` 仅管本校区；`system_admin` 看全局总览与汇总报表。
- 解决问题：保证本地管理效率与全局监管能力并存。
- 验证指标：越权拦截率 100%，权限边界测试全部通过。


## 8.2 高并发落点（穿插实现，详细版）

1. 幂等键防重复写（入口层）
- 实现方式：关键写接口要求 `Idempotency-Key`；同键同参返回同结果，不重复落库。
- 适用场景：资产申报、日报提交、预约创建。
- 验证指标：重复请求重复写入率接近 0；幂等命中返回一致。

2. 分布式锁防热点冲突（竞争层）
- 实现方式：Redis 锁保护热点资源（如“预算池行”“同实验室同时间段”）。
- 适用场景：并发锁额度、并发抢占时段。
- 验证指标：冲突数据为 0；高并发下无超额、无双写成功。

3. 事务与索引保证一致性（持久层）
- 实现方式：事务内完成“校验-扣减/转换-写流水”；配合唯一索引和状态机约束。
- 适用场景：`locked -> used`、`locked -> release`、审批流转。
- 验证指标：预算账目平衡（总额=可用+锁定+已用）、无脏数据、无非法状态跳转。

4. 异步解耦缩短主链路（处理层）
- 实现方式：通知、汇总、非关键日志改为消息异步处理。
- 适用场景：提交后通知管理员、跨校区总表更新。
- 验证指标：主请求 P95/P99 明显下降；队列积压可恢复；失败可重试补偿。

5. 缓存与限流保护系统（稳定层）
- 实现方式：高频查询缓存；写接口按用户/接口限流；过载时降级返回“处理中”。
- 适用场景：管理员集中查看日报、统计面板高频刷新。
- 验证指标：峰值期间错误率可控、数据库 QPS 不击穿、服务可用性保持目标值。

## 9. 分阶段实施顺序（建议）

1. 阶段A：数据库与后端基础  
   - 资产表/日报表建模、迁移脚本、服务层事务骨架。  
2. 阶段B：SeaweedFS 接入  
   - 配置、上传服务封装、资产图片与日报图片接入。  
3. 阶段C：业务接口落地  
   - 资产申报/审批/入库接口，日报提交/审核接口。  
4. 阶段D：小程序页面落地  
   - 学生日报上传、管理员查看审核。  
5. 阶段E：测试与论文数据  
   - 并发压测、故障测试、结果对比图表。
