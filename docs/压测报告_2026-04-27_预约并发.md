# 预约接口并发压测报告（2026-04-27）

## 1. 压测目标

本次压测聚焦毕设题目中的“高并发控制”和“分布式一致性控制”两点：

1. 在高并发请求下，预约写入是否出现重复占用。
2. 幂等机制（`Idempotency-Key`）是否有效拦截重复提交。
3. 分布式锁链路在开启/关闭时的行为差异，为阶段 C（多实例部署）提供改造依据。

## 2. 测试环境与参数

1. 后端：Flask（本地单实例，`127.0.0.1:5000`）。
2. 压测脚本：`benchmark/reservation_concurrency_benchmark.py`。
3. 并发参数：`concurrency=80`，`total=400`。
4. 测试账号：`student1`。
5. 目标实验室：`campus_id=5`，`lab_id=17`。
6. 预约日期：`2026-04-30`（分场景使用不同时段，避免相互污染）。

## 3. 场景一：幂等模式对比（隔离时段）

数据来源：

1. `benchmark/results/reservation_none_isolated_20260427_233145.json`
2. `benchmark/results/reservation_same_isolated_20260427_233149.json`
3. `benchmark/results/reservation_unique_isolated_20260427_233154.json`

| 场景 | 目标时段 | 吞吐 (RPS) | P95 延迟 (ms) | HTTP 分布 | 业务码分布 | 返回记录数 | 唯一预约 ID 数 |
|---|---|---:|---:|---|---|---:|---:|
| `idempotency=none` | 11:00-12:00 | 106.69 | 910.75 | 200:5, 400:395 | 0:5, 40000:395 | 5 | 5 |
| `idempotency=same` | 13:00-14:00 | 111.88 | 732.93 | 200:381, 409:19 | 0:381, 40924:19 | 381 | 1 |
| `idempotency=unique` | 15:00-16:00 | 56.05 | 2731.89 | 200:6, 400:394 | 0:6, 40000:394 | 6 | 6 |

### 关键结论

1. `same` 模式下 `unique_reservation_ids=1`，说明幂等键对“重复请求回放同一结果”生效。
2. `none` 与 `unique` 模式下出现多条成功写入（5/6 条），说明在当前部署条件下仍存在并发竞态窗口。
3. `unique` 模式 P95/P99 显著变高，说明高竞争下冲突处理和回滚开销明显。

## 4. 场景二：分布式锁开关对比（阶段 C 前置验证）

数据来源：

1. `benchmark/results/reservation_unique_lock0_20260427_233251.json`
2. `benchmark/results/reservation_unique_lock1_20260427_233301.json`

| 场景 | 目标时段 | 吞吐 (RPS) | P95 延迟 (ms) | HTTP 分布 | 业务码分布 | 写入结果 |
|---|---|---:|---:|---|---|---|
| `ENABLE_DISTRIBUTED_LOCK=0` | 17:00-18:00 | 55.71 | 2821.42 | 200:6, 400:394 | 0:6, 40000:394 | 仍有重复占用风险 |
| `ENABLE_DISTRIBUTED_LOCK=1` | 19:00-20:00 | 62.78 | 2193.06 | 500:400 | 50041:400 | 分布式锁链路不可用 |

### 关键结论

1. 当前代码中分布式锁开启后强依赖 Redis；未配置 Redis 时统一返回 `50041`（“分布式锁已启用但 Redis 不可用”）。
2. 这说明阶段 C 必须落地 Redis（或降级锁策略），否则多实例场景不可用。

## 5. 本次压测中修复/校准项

为保证报告有效性，压测前对脚本做了两处校准：

1. 登录令牌兼容：同时支持后端返回 `token` 或 `access_token`。
2. `idempotency=same` 时保证请求体一致（避免因 `purpose` 字段变化触发 40921 假冲突）。

## 6. 对阶段 C 的直接输入

基于本次压测，阶段 C 需要优先完成：

1. Redis 基础设施落地（容器化），确保分布式锁可用。
2. 多实例部署 + Nginx 负载均衡，验证跨实例并发一致性。
3. 压测脚本纳入 CI/验收流程，补充“多实例下同窗冲突为 0”的验收门槛。

## 7. 验收建议（答辩可展示）

1. 展示 `same` 模式：400 并发下只对应 1 个唯一预约 ID。
2. 展示 `lock=0` 与 `lock=1` 对比：说明“仅写代码”不等于“链路可用”，基础设施是分布式能力的一部分。
3. 展示阶段 C 完成后的复测：目标是同窗预约成功数稳定为 1，冲突返回可控，无 5xx。
