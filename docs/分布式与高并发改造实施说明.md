# 分布式与高并发改造实施说明（毕设项目）

## 1. 文档目标

本说明用于明确本项目中：

1. 分布式能力体现在哪里。
2. 高并发能力体现在哪里。
3. 如何在代码与部署中证明这些能力已落地。
4. 后续代码改造的实施顺序与验收标准。

项目名称：支持跨校区协作的分布式实验室管理系统设计与实现

---

## 2. 当前系统现状（改造前）

### 2.1 已有基础（可复用）

1. 多校区数据模型已具备：`campus -> lab -> reservation`。
2. 权限按角色和校区隔离：`student/teacher/lab_admin/system_admin`。
3. 预约、审批、统计、日志链路完整，业务规则清晰。

### 2.2 当前短板（必须补齐）

1. 预约创建是“先查冲突再写入”，高并发下存在竞态窗口。
2. 没有分布式锁、幂等机制、消息削峰机制。
3. 无 Redis 缓存与热点保护，所有请求直打数据库。
4. 部署层未体现多实例、负载均衡、异步解耦。

---

## 3. 分布式能力如何体现

本项目的“分布式”不只体现在“多台机器”，而是体现在“跨节点协同 + 故障隔离 + 数据一致性策略”。

### 3.1 多实例无状态服务 + 统一入口

体现点：

1. Flask 服务改为无状态（会话仅用 JWT，不存本地内存）。
2. 至少部署 2 个后端实例。
3. Nginx 作为统一入口并做负载均衡。

证明方式：

1. 压测时两个实例均有请求命中日志。
2. 下线一个实例后，系统仍可用。

### 3.2 跨校区协同的分布式资源治理

体现点：

1. 全局统一预约接口，校区作为资源维度而非系统边界。
2. `lab_admin` 仅处理本校区，`system_admin` 可跨校区调度。
3. 统计支持校区聚合和全局聚合。

证明方式：

1. 演示海淀/丰台/海南校区同时预约与审批。
2. 演示全局管理员查看跨校区总览与排行。

### 3.3 分布式一致性控制（预约核心）

体现点：

1. Redis 分布式锁控制同一实验室同一时间片并发写入。
2. 请求幂等键（Idempotency-Key）避免重复提交。
3. 数据库唯一约束/索引与事务作为最终一致性兜底。

证明方式：

1. 并发提交同一时段，只允许 1 条成功占用。
2. 同一请求重试返回同一业务结果，不重复创建预约。

### 3.4 异步解耦（分布式消息）

体现点：

1. 预约成功后通知、统计增量、审计扩展走消息队列异步消费。
2. 核心交易链路缩短，非关键逻辑解耦。

证明方式：

1. 主请求延迟下降。
2. 消费积压可观测，失败可重试。

## 4. 高并发能力如何体现

### 4.1 预约入口削峰与限流

体现点：

1. 网关限流（IP/用户维度）。
2. 应用层限流（按用户、按实验室）。
3. 排队机制（超阈值请求进入异步队列）。

证明方式：

1. 突发流量下错误率可控，不出现整体雪崩。
2. 系统返回“处理中”而非超时崩溃。

### 4.2 热点资源并发争抢控制

体现点：

1. Redis 原子检查（Lua）+ 分布式锁。
2. DB 二次校验与事务提交。
3. 审批通过同样走冲突防重逻辑，避免“审批时超卖”。

证明方式：

1. 100/500/1000 并发下，冲突数据为 0。
2. 不出现同实验室同时间段重复占用。

### 4.3 缓存加速与数据库保护

体现点：

1. 实验室列表、排期查询走缓存。
2. 缓存失效使用随机过期 + 互斥重建，防击穿。
3. 热点统计接口缓存化，减少聚合 SQL 压力。

证明方式：

1. 查询接口 P95 显著下降。
2. DB QPS 在高峰期受控。

   2. 增加 Redis 分布式锁与原子冲突校验。
   3. 审批通过逻辑补并发安全校验。

   1. 接收 `Idempotency-Key` 请求头。

   1. 增加版本号/幂等关联字段（必要时）。

### 5.2 前端与小程序改造

1. `frontend/api/request.js`

### 5.3 基础设施与压测资产

建议新增目录：

2. `infra/docker/`：后端实例、Redis、MQ 编排。

## 6. 数据一致性与事务策略

预约写入策略：

1. 先校验参数与权限。
2. 进入分布式锁（按 `lab_id + date + time_slot`）。
3. Redis/DB 冲突检查。
4. DB 事务写入预约与日志。
5. 提交后异步投递通知与统计消息。

失败处理策略：

1. 锁冲突：快速失败并提示重试。
2. 幂等命中：返回历史成功结果。
3. DB 冲突：事务回滚并返回可读错误码。

## 7. 可观测性与答辩展示点

### 7.1 指标

1. 吞吐量（TPS/QPS）。
2. 延迟（P50/P95/P99）。
3. 错误率（4xx/5xx）。
4. 锁竞争率、队列积压长度、缓存命中率。

### 7.2 日志与追踪

1. 每个请求附 `trace_id`。
2. 预约链路记录：入参校验 -> 锁 -> 冲突检查 -> 提交 -> 异步投递。
3. 异常分类：业务冲突、限流、系统错误。

### 7.3 答辩演示建议

1. 并发抢同一时段：展示只成功 1 条。
2. 重复提交：展示幂等生效。
3. 下线一个后端实例：展示服务可用。
4. 展示改造前后压测对比图表。

## 8. 分阶段实施计划

### 阶段 A（先落地，1-2 天）

1. 幂等键 + 分布式锁 + 冲突原子校验。
2. 预约与审批并发安全闭环。
3. 基础压测脚本与第一版报告。

### 阶段 B（增强，2-4 天）

1. Redis 缓存排期/列表/统计。
2. 限流策略与错误码规范化。
3. 前端/小程序提交态改造。

### 阶段 C（答辩强化，2-3 天）

1. MQ 异步通知与统计解耦。
2. 双实例 + Nginx 演示环境。
3. 完整实验数据、图表、结论整理入论文/PPT。

---

## 9. 验收标准（必须满足）

1. 高并发同窗预约无重复占用，数据冲突为 0。
2. 重复请求不产生重复预约。
3. 单实例故障不影响核心预约与查询。
4. 压测下系统错误率和延迟指标达标。
5. 论文与答辩材料中可展示架构图、流程图、对比实验结果。

---

## 10. 结论

本项目后续改造将以“预约链路并发安全”为核心，逐步补齐“分布式协同、异步解耦、缓存防护、限流削峰”四大能力。  
完成后可在代码、部署、压测和答辩展示四个层面清晰证明题目中的“分布式 + 高并发”要求。
