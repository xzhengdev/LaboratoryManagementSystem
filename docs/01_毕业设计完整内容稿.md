# 实验室预约管理系统（毕业设计）完整内容稿

## 1. 课题目标与研究意义

### 1.1 课题背景
高校实验室资源分散在不同校区，人工登记与电话沟通效率低，容易出现预约冲突、审批滞后和利用率不透明等问题。为提升实验室资源配置效率，设计并实现一套支持多角色、多端协同的实验室预约管理系统。

### 1.2 课题目标
本课题目标是实现一个可运行、可演示、可扩展的实验室预约系统，完成以下核心能力：
- 支持学生、教师、实验室管理员、系统管理员多角色协同。
- 支持校区、实验室、设备、预约、审批、统计等完整业务闭环。
- 支持 Web/H5 与微信小程序双端访问。
- 引入智能问答助手，提升查询与引导效率。

### 1.3 研究意义
- 实践意义：减少人工协调成本，提高实验室利用效率。
- 教学意义：为教学管理数字化提供可参考方案。
- 工程意义：形成从需求到设计、实现、测试、部署的完整毕业设计流程。

## 2. 需求分析

### 2.1 用户角色需求
- 学生：查询实验室、提交预约、查看与取消本人预约。
- 教师：与学生类似，支持教学相关预约。
- 实验室管理员：管理本校区实验室/设备，处理预约审批。
- 系统管理员：跨校区管理用户、校区、实验室、全局审批与统计。

### 2.2 功能需求
- 用户认证与权限控制。
- 校区管理（增删改查、封面图）。
- 实验室管理（信息维护、开放时间、容量、照片）。
- 设备管理（按实验室维度维护状态与库存）。
- 预约管理（创建、冲突检测、状态流转、取消）。
- 审批管理（待审批列表、通过/驳回、审批意见）。
- 统计分析（总览、校区维度、实验室利用率）。
- Agent 问答（我的预约、可预约实验室、排期与统计查询）。

### 2.3 非功能需求
- 可用性：界面清晰，核心操作步骤不超过 3~4 步。
- 安全性：JWT 鉴权、角色权限校验、关键操作日志。
- 性能：常规查询接口在教学场景下保持可接受响应。
- 可维护性：后端分层（API/Service/Model/Utils）与文档化。
- 可扩展性：支持规则问答与 LLM 模式切换。

### 2.4 核心业务规则
- 预约时间必须合法：开始时间早于结束时间。
- 不允许预约过去时间段。
- 预约开始时间需至少提前 30 分钟。
- 预约日期限制在未来 7 天内。
- 预约时间必须在实验室开放时段内。
- 参与人数不可超过实验室容量。
- 活跃预约（pending/approved）时间段不可冲突。
- 学生/教师预约默认待审批，管理员预约可自动通过。

## 3. 总体设计

### 3.1 系统架构
系统采用前后端分离架构：
- 后端：Flask 提供 RESTful API。
- 前端：uni-app 提供 Web/H5 管理与用户端。
- 小程序：微信原生小程序实现移动端场景。
- 数据层：开发环境 SQLite，生产可切换 MySQL。

### 3.2 模块划分
- 认证模块
- 校区模块
- 实验室模块
- 设备模块
- 预约模块
- 审批模块
- 统计模块
- Agent 模块

### 3.3 数据库设计说明
核心实体：
- User（用户）
- Campus（校区）
- Laboratory（实验室）
- Equipment（设备）
- Reservation（预约）
- Approval（审批）
- OperationLog（操作日志）

关键关系：
- 校区与实验室：一对多
- 实验室与设备：一对多
- 用户与预约：一对多
- 预约与审批：一对多（审批记录）

### 3.4 接口设计原则
- 统一前缀：`/api/*`
- 统一返回结构：`code/message/data`
- 统一鉴权方式：`Authorization: Bearer <token>`
- 统一错误码便于前端处理与排错

## 4. 详细设计

### 4.1 登录与鉴权流程
1. 用户输入账号、密码、角色。
2. 后端校验通过后签发 JWT。
3. 客户端存储 token 并在后续请求头携带。
4. 401 时清理会话并跳转登录页。

### 4.2 预约流程
1. 用户选择校区与实验室。
2. 填写日期、时间段、用途、参与人数。
3. 后端执行参数校验、时间校验、容量校验、冲突校验。
4. 根据角色生成预约状态（pending/approved）。
5. 返回预约结果并写入操作日志。

### 4.3 审批流程
1. 管理员查看待审批列表。
2. 对预约执行通过或拒绝并填写意见。
3. 通过时再次进行冲突校验，避免等待期冲突。
4. 写入审批记录并更新预约状态。

### 4.4 统计流程
- 聚合预约数据，输出总览指标、校区统计、实验室利用率。
- 为管理决策和答辩展示提供数据支撑。

### 4.5 Agent 流程
- 入口：`POST /api/agent/chat`
- 模式：规则模式（默认）+ 可选 LLM 模式。
- 能力：查询我的预约、可预约实验室、指定排期、统计信息。

## 5. 系统实现

### 5.1 后端实现
- 技术：Flask + SQLAlchemy + Flask-Migrate + JWT + CORS。
- 分层：API 层负责路由，Service 层封装业务规则，Model 层映射数据，Utils 层提供通用能力。
- 已实现预约核心规则与权限边界控制。

### 5.2 前端实现（uni-app）
- 采用页面与组件分离。
- 封装统一请求模块，自动注入 token。
- 通过 `navigation` 配置维护多角色导航结构。

### 5.3 小程序实现
- 封装 `wx.request` 网络层与会话层。
- 实现登录、实验室浏览、预约、我的预约、个人中心与 Agent 页面。
- 保持与后端 API 兼容。

### 5.4 关键技术点
- 冲突检测算法：`start < existing_end && end > existing_start`
- 权限控制：登录鉴权 + 角色鉴权 + 数据归属二次校验
- 多端复用：统一 API 约定，端内封装差异

## 6. 测试与结果分析

### 6.1 测试目标
验证系统功能完整性、业务规则正确性、权限边界有效性。

### 6.2 测试内容
- 功能测试：登录、预约、审批、统计、个人中心。
- 业务规则测试：时间限制、容量限制、冲突校验。
- 权限测试：不同角色访问与操作边界。
- 接口测试：错误码与异常处理一致性。

### 6.3 结果说明（可按实际补图）
- 核心流程可正常运行：登录 -> 预约 -> 审批 -> 查询。
- 关键规则校验有效：冲突预约被拦截，权限越权被拒绝。
- 多端联调可完成基础演示。

### 6.4 已知问题与改进方向
- 自动化测试覆盖不足（后续补充 pytest 与接口回归）。
- 生产安全配置需加强（密钥、CORS 白名单、日志治理）。
- Agent 路由跳转映射可进一步统一。

## 7. 部署与运行说明

### 7.1 开发环境
- Python 3.x
- Flask 后端运行于 `backend/`
- uni-app 前端运行于 `frontend/`
- 微信开发者工具打开 `miniprogram/`

### 7.2 启动步骤
1. 启动后端并初始化数据。
2. 启动前端 H5 或打开小程序。
3. 使用演示账号登录并完成预约审批流程演示。

### 7.3 生产部署建议
- 后端：Gunicorn + Nginx
- 数据库：MySQL
- 前端：静态资源部署至 Nginx
- 反向代理：统一 `/api` 转发
