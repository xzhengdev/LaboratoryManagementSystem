# 实验室预约管理系统需求分析文档

## 1. 引言

### 1.1 编写目的
本文档用于明确“实验室预约管理系统”的业务目标、用户需求、功能需求与非功能需求，为系统设计、编码实现、测试验收和毕业论文撰写提供依据。

### 1.2 项目背景
高校实验室通常分布于多个校区，传统预约方式依赖线下登记或人工沟通，存在预约冲突、审批延迟、资源利用率不透明等问题。为提升实验室资源调度效率与管理规范性，建设本系统。

### 1.3 术语说明
- 预约：用户提交实验室在某日期与时段的使用申请。
- 审批：管理员对预约申请执行通过或驳回。
- 活跃预约：状态为 `pending`（待审批）或 `approved`（已通过）的预约。
- Agent：系统内智能问答助手，支持规则模式和可选 LLM 模式。

---

## 2. 需求概述

### 2.1 系统目标
- 提供多角色统一预约平台，支持学生、教师、实验室管理员、系统管理员协同。
- 建立“预约申请-审批-结果反馈”的完整业务闭环。
- 支持多端访问（Web/H5 与微信小程序）。
- 提供数据统计与智能问答能力，提高管理效率和使用体验。

### 2.2 用户角色
- 学生（student）
- 教师（teacher）
- 实验室管理员（lab_admin）
- 系统管理员（system_admin）

### 2.3 运行环境约束
- 后端：Flask API 服务。
- 数据库：开发环境 SQLite，生产可切换 MySQL。
- 客户端：uni-app Web/H5、微信小程序。

---

## 3. 业务流程需求

### 3.1 登录鉴权流程
1. 用户输入账号、密码、角色。
2. 系统校验身份信息并签发 JWT。
3. 客户端在后续请求头中携带 Token。
4. Token 失效时引导重新登录。

### 3.2 预约流程
1. 用户浏览校区与实验室信息。
2. 填写预约日期、开始时间、结束时间、用途、参与人数。
3. 系统执行参数校验、时间校验、容量校验、冲突校验。
4. 学生/教师预约进入待审批；管理员预约可直接通过。
5. 系统返回预约结果并记录日志。

### 3.3 审批流程
1. 管理员查看待审批预约列表。
2. 管理员选择“通过”或“驳回”并填写审批意见。
3. 若通过，系统再次校验时间冲突。
4. 系统更新预约状态并生成审批记录。

### 3.4 查询与统计流程
- 用户可查看“我的预约”及预约详情。
- 管理员可查看统计总览、校区统计、实验室利用率。

### 3.5 智能助手流程
- 用户输入自然语言问题。
- Agent 识别意图并返回相关数据与操作建议。
- 支持“我的预约、可预约实验室、排期、统计”等查询。

## 4. 功能需求

### 4.1 用户与认证模块
- 支持账号密码登录。
- 支持按角色登录。
- 支持获取与更新个人资料。
- 支持头像上传。
- 支持登录态失效处理。

### 4.2 校区管理模块
- 校区列表与详情查询。
- 校区新增、编辑、删除（系统管理员）。
- 校区封面图上传。

### 4.3 实验室管理模块
- 实验室列表、详情、排期查询。
- 实验室新增、编辑、删除。
- 实验室照片上传/删除。
- 删除实验室时需进行关联预约校验。

### 4.4 设备管理模块
- 设备列表查询（可按实验室筛选）。
- 设备新增、编辑、删除。
- 设备状态与数量维护。
- 实验室管理员仅可管理本校区数据。

### 4.5 预约管理模块
- 创建预约。
- 取消预约。
- 查看我的预约。
- 查看预约详情。
- 管理端预约列表查询。

### 4.6 审批管理模块
- 待审批列表查询。
- 审批通过。
- 审批驳回。
- 审批意见记录与审批历史展示。

### 4.7 统计分析模块
- 总览统计（预约量、通过量等关键指标）。
- 校区维度统计。
- 实验室维度利用率统计。

### 4.8 Agent 智能助手模块
- 对话入口与消息处理。
- 规则模式问答。
- 可选 LLM 模式问答。
- 返回查询结果与建议动作。

## 5. 核心业务规则需求

### 5.1 预约规则
- 必填字段：`campus_id`、`lab_id`、`reservation_date`、`start_time`、`end_time`、`purpose`。
- 开始时间必须早于结束时间。
- 不允许预约过去日期。
- 预约日期不得超过未来 7 天。
- 当天预约需满足“开始时间至少晚于当前时间 30 分钟”。
- 预约时间必须在实验室开放时段内。
- 参与人数不得超过实验室容量。
- 活跃预约间不得时间重叠。

### 5.2 权限与数据范围规则
- 所有敏感接口必须登录。
- 学生/教师默认仅可访问本人预约数据。
- 实验室管理员仅可操作本校区相关数据。
- 系统管理员可跨校区管理。

### 5.3 审批规则
- 仅管理员可审批。
- 仅 `pending` 状态可被审批。
- 审批通过前需再次冲突检测。

### 5.4 状态流转规则
- 预约状态：`pending` -> `approved/rejected`，可转 `cancelled`。
- 已取消或已拒绝预约不可重复取消。

## 6. 非功能需求

### 6.1 安全性
- JWT 鉴权与角色授权。
- 关键操作日志记录。
- 上传大小限制与非法请求拦截。
- 生产环境需替换默认密钥并限制跨域来源。

### 6.2 性能与容量
- 常规查询接口应满足教学场景并发访问。
- 列表查询与统计接口响应时间应保持在可接受范围。
- 支持后续通过索引优化与缓存提升性能。

### 6.3 可用性
- 页面操作流程清晰，关键任务步骤简洁。
- 错误提示明确，便于用户纠错。
- 支持多端一致的核心业务体验。

### 6.4 可维护性
- 后端采用分层架构，业务规则集中在 Service 层。
- 接口响应结构统一，便于前端与测试联调。
- 关键模块具备文档说明，便于后续扩展。

### 6.5 可扩展性
- 数据库支持 SQLite 到 MySQL 的平滑切换。
- Agent 模块支持从规则模式扩展到第三方 LLM。

---

## 7. 接口需求

### 7.1 接口风格
- RESTful 风格。
- 接口统一前缀：`/api`。
- 认证方式：`Authorization: Bearer <token>`。

### 7.2 统一返回规范
- 成功：`{ code: 0, message: "success", data: ... }`
- 失败：`{ code: 非0, message: "错误信息", data: null }`

### 7.3 主要接口范围
- 认证：`/api/auth/*`
- 校区：`/api/campuses*`
- 实验室：`/api/labs*`
- 设备：`/api/equipment*`
- 预约：`/api/reservations*`
- 审批：`/api/approvals*`
- 统计：`/api/statistics*`
- Agent：`POST /api/agent/chat`

## 8. 数据需求

### 8.1 核心实体
- 用户（User）
- 校区（Campus）
- 实验室（Laboratory）
- 设备（Equipment）
- 预约（Reservation）
- 审批（Approval）
- 操作日志（OperationLog）

### 8.2 关键数据约束
- 实验室归属唯一校区。
- 预约必须关联有效用户和实验室。
- 审批记录必须关联有效预约。
- 上传资源路径需可追溯并可访问。

## 9. 验收需求

### 9.1 功能验收
- 能完成登录、预约、审批、统计、个人信息维护、Agent 问答等核心功能演示。
- 多角色权限符合预期，不可越权访问。

### 9.2 规则验收
- 非法时间段、超容量、冲突预约应被正确拦截。
- 审批状态流转正确，审批记录完整。

### 9.3 多端验收
- Web/H5 与小程序端均可完成核心业务流程。
- 两端接口调用与错误处理行为一致。

### 9.4 文档验收
- 需求、设计、测试文档齐全并与实现一致。

---

## 10. 风险与边界

### 10.1 当前边界
- 以教学与毕业设计演示场景为主，优先保障核心业务闭环。
- 暂不覆盖企业级高并发与复杂运维体系。

### 10.2 主要风险
- 自动化测试不足可能引入回归风险。
- 生产环境安全配置不完善可能导致安全隐患。
- Agent 在复杂语义场景下可能存在理解偏差。

### 10.3 应对策略
- 优先补充预约/审批/权限核心测试用例。
- 上线前完成密钥替换、CORS 白名单与日志审计配置。
- 对 Agent 输出增加兜底提示与人工引导。
