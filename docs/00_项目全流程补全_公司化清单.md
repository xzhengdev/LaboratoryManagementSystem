# 实验室预约项目全流程补全（公司化版本）

> 适用对象：当前 `lab` 项目（Flask 后端 + uni-app 前端 + 微信小程序）
> 目标：把“能演示”升级为“可交付、可维护、可上线”的完整项目流程

## 1. 当前项目现状（基于仓库）

已具备：
- 业务主流程：登录、校区/实验室/设备、预约、审批、统计、Agent
- 多端形态：后端 API、H5/管理端、小程序端
- 角色权限：student/teacher/lab_admin/system_admin
- 基础文档：README、后端说明、前端架构说明、Agent 设计文档

主要缺口（工程化角度）：
- 缺统一 PRD/范围边界文档（需求优先级、验收口径未完全固化）
- 缺 API 契约文档（OpenAPI/字段级约束）
- 缺自动化测试体系（单测/接口回归/E2E）
- 缺 CI/CD 与发布流程文档
- 缺运维与安全基线（日志规范、告警、备份、密钥轮换、审计）

## 2. 公司项目的完整流程（端到端）

### 阶段 A：立项与目标对齐（0-1 周）

交付物：
- 项目章程（Project Charter）
- 北极星指标与业务 KPI（如预约成功率、审批时长、活跃用户数）
- 范围定义（In Scope / Out of Scope）
- 干系人清单与职责（RACI）

你这个项目建议补充：
- 明确“毕业设计演示版”与“可上线版”边界
- 明确优先服务对象（学生/教师/管理员）与核心场景优先级

### 阶段 B：需求分析（1-2 周，前期核心）

交付物：
- BRD（业务需求文档）
- PRD（产品需求文档）
- 用户故事（User Story）+ 验收标准（Acceptance Criteria）
- 需求优先级（MoSCoW：Must/Should/Could/Won't）
- 业务规则清单（尤其预约冲突、审批、权限）

你提到的“开发需求”，在公司里至少要包含：
- 功能需求：每个模块要做什么
- 非功能需求：性能、安全、可用性、可扩展性
- 数据需求：核心实体、字段约束、状态机
- 合规需求：日志留痕、权限隔离、隐私字段处理
- 验收需求：什么叫“做完”

### 阶段 C：方案设计与技术预研（并行 1 周）

交付物：
- 架构设计（逻辑架构 + 部署架构）
- 技术选型对比（为什么 Flask/uni-app/小程序）
- 数据库设计（ER 图、索引策略、迁移策略）
- API 设计文档（建议 OpenAPI）
- 风险清单（Risk Register）与预案

针对当前项目建议补：
- `OpenAPI` 文档（认证、预约、审批、统计、Agent）
- 预约状态流转图（pending/approved/rejected/cancelled）
- 多端路由映射规范（Web 路径与小程序路径统一策略）

### 阶段 D：项目计划与研发准备（0.5-1 周）

交付物：
- 迭代计划（Sprint Backlog）
- WBS（任务分解）+ 工时评估
- 分支策略（Git Flow 或 Trunk Based）
- 代码规范（Lint、命名、提交规范、Review 清单）
- 环境规划（dev/test/staging/prod）

当前仓库建议补：
- `CONTRIBUTING.md`（分支、提交流程、Review 要求）
- Issue/PR 模板
- 环境变量分层规范（`.env.dev/.env.test/.env.prod`）

### 阶段 E：研发实施（2-4 周，迭代式）

执行方式：
- 后端、前端、小程序按迭代并行
- 每个需求都带“验收标准 + 测试用例”
- 每日站会跟踪风险与阻塞

交付物：
- 可运行增量版本
- 变更记录（Changelog）
- 接口变更同步通知

### 阶段 F：测试与验收（1-2 周）

交付物：
- 测试计划与测试报告
- 缺陷清单（严重级别 + 关闭标准）
- UAT 验收记录

最少要覆盖：
- 单元测试：预约冲突判定、权限边界、审批逻辑
- 接口测试：鉴权、参数校验、错误码一致性
- 端到端流程：登录 -> 预约 -> 审批 -> 结果查询
- 回归测试：每次改动后核心链路不回退

### 阶段 G：发布上线（0.5 周）

交付物：
- 发布清单（Release Checklist）
- 回滚预案（数据库和应用双回滚）
- 上线公告与变更窗口

你这个项目建议：
- 明确 Nginx 反向代理和静态资源部署步骤
- 明确数据库备份与恢复演练步骤

### 阶段 H：运维监控与持续优化（长期）

交付物：
- 监控看板（可用性、错误率、响应时间）
- 告警策略（接口 5xx、登录失败率突增等）
- 周/月复盘报告

可落地项：
- 结构化日志 + 请求追踪 ID
- 操作审计报表（管理员关键操作）
- Agent 答非所问率与转人工策略

## 3. 前期工作到底要做哪些（你现在最需要补的）

按优先级排序：

P0（必须先做）
- 需求基线：PRD + 范围边界 + 验收标准
- API 契约：统一字段、错误码、示例请求响应
- 数据与状态机：预约/审批状态图 + 关键约束
- 测试基线：核心链路测试用例（先手工后自动化）

P1（强烈建议）
- 迭代计划与里程碑
- 代码协作规范（分支、提交、Review）
- 安全基线（密钥、权限、CORS、上传限制）
- 发布与回滚 SOP

P2（增强项）
- CI/CD 自动化
- 性能压测与容量评估
- 监控告警与运维看板
- 用户反馈闭环（埋点与行为分析）

## 4. 结合本项目的“补全版里程碑”

M1（本周）
- 完成 PRD v1 + API 契约 v1 + 测试用例 v1
- 输出预约/审批状态机与权限矩阵

M2（下周）
- 补齐核心自动化测试（预约、审批、权限）
- 建立发布清单和回滚文档

M3（第 3 周）
- 完成 CI 基线（至少 lint + backend 测试）
- 建立日志与告警最小闭环

M4（第 4 周）
- 完成一轮 UAT + 问题修复
- 输出可答辩/可交付版本报告
