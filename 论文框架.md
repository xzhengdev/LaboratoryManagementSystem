## 摘要 / Abstract

### 中文摘要写作要点

随着高校多校区办学与实验教学规模扩大，实验室资源在空间分布、开放时段、设备状态和审批流程方面呈现出更高的管理复杂度。针对传统实验室管理中存在的预约效率低、跨校区资源协同不足、审批流程分散、数据统计困难等问题，本文设计并实现了一个支持跨校区协作的分布式实验室管理系统。系统采用 Flask 构建后端 API 服务，基于 SQLAlchemy 完成数据建模，使用 JWT 实现身份认证与权限控制，并提供 Web 管理端和微信小程序端两类访问入口。系统主要实现用户管理、校区管理、实验室管理、设备管理、预约冲突检测、审批流转、统计分析与智能助手等功能。测试结果表明，系统能够较好地支撑多角色、多校区、多终端的实验室预约与管理需求，提高实验室资源使用效率和管理信息化水平。

关键词：实验室管理；跨校区协作；预约系统；权限控制；智能助手

### Abstract 写作要点

With the expansion of multi-campus university operation and experimental teaching, laboratory resources become more difficult to manage in terms of spatial distribution, opening hours, equipment status and approval procedures. To address the problems of inefficient manual reservation, insufficient cross-campus collaboration, decentralized approval and weak statistical support, this thesis designs and implements a distributed laboratory management system supporting cross-campus collaboration. The system uses Flask to build backend APIs, SQLAlchemy for data modeling, JWT for authentication and authorization, and provides both a Web management portal and a WeChat Mini Program client. The system implements user management, campus management, laboratory management, equipment management, reservation conflict detection, approval workflow, statistical analysis and an intelligent assistant. Test results show that the system can support multi-role, multi-campus and multi-terminal laboratory reservation scenarios, improving resource utilization and the informatization level of laboratory management.

Keywords: laboratory management; cross-campus collaboration; reservation system; access control; intelligent assistant

## 第1章 绪论

### 1.1 课题背景与研究意义

随着高等教育信息化建设持续推进，信息技术已经成为高校治理能力提升和教学资源优化配置的重要支撑。《教育信息化 2.0 行动计划》指出，教育信息化能够突破时空限制，促进教育资源共享，并提升教育治理与科学决策能力[1]。在高校实验教学场景中，实验室既承担课程教学、实践训练和科研支持任务，也是培养学生工程实践能力、创新能力的重要空间。近年来，教育部陆续发布高校实验室安全与消防安全相关规范，进一步强调高校需要建立制度化、规范化、信息化的实验室管理体系[2][3]。

在多校区办学模式下，实验室管理的复杂度进一步增加。不同校区之间实验室数量、设备类型、开放时间、管理人员和使用群体存在差异，传统依赖线下登记、人工审批和分散表格维护的方式容易造成信息不透明、预约冲突、资源闲置与管理追踪困难等问题。已有研究指出，高校实验室预约管理系统能够通过在线预约、预约查询、数据统计等功能提升实验室管理规范性和信息化程度[4]；基于低代码平台的实验室预约系统也表明，在线预约能够改善人工操作低效问题并提高实验室与设备利用率[5]。因此，面向跨校区协作场景构建统一的实验室管理系统，具有明确的现实需求。

本课题的研究意义主要体现在三个方面。第一，在应用层面，系统可将校区、实验室、设备、预约和审批信息集中管理，降低师生预约成本，减少人工协调工作量。第二，在管理层面，系统通过多角色权限划分、审批流转和操作记录，为实验室资源使用提供可追踪、可统计、可审计的数据支撑。第三，在技术层面，系统采用“统一后端服务 + Web 管理端 + 微信小程序端 + 智能助手”的架构，将跨校区资源协同、权限控制、冲突检测和自然语言交互结合起来，为高校实验室管理数字化提供一个可实现、可扩展的工程方案。

### 1.2 国内外研究现状

国外关于实验室预约与共享资源调度的研究较早展开。Ma 对开放实验室与移动端预约系统进行了研究，指出移动终端能够使教师和学生更便捷地查询实验室资源并提交预约申请，从而提升实验室开放管理效率[6]。Khazri 等针对远程实验室提出预约系统与排队机制，强调共享实验资源需要通过调度机制降低等待时间并提高资源利用率[7]。Wang 等从高校实验设备预约管理角度提出基于 B/S 架构的系统方案，说明模块化和流程化设计有助于提高实验室设备管理的标准化水平[8]。Zhu 等在实验室预约系统中引入 RFID 技术，用于加强实验过程中的实时监测和自动化管理[9]。

智慧校园研究为实验室管理系统提供了更广阔的背景。Li 从物联网角度设计智慧校园管理系统，提出通过统一数据采集和后台管理提升校园管理质量[10]。Zhang 等对智慧校园技术和应用进行系统综述，认为智慧校园是教育系统智能化发展的重要形态，涉及教学、管理、服务、能源等多个领域[11]。这些研究说明，实验室预约管理不应只被视为单一业务系统，而应纳入高校智慧校园和资源治理体系中整体设计。

国内相关研究主要集中在高校实验室信息化、预约管理和前后端分离系统实现。李丹设计的基于 B/S 的高校实验室预约管理系统支持专业实验室、教室和会议室预约，并与统一身份认证系统对接，提高了实验室管理规范性[4]。杨萍萍、白艳茹基于低代码平台实现高校实验室预约系统，为实验室开放预约和资源利用率提升提供了实践案例[5]。王以伍、舒晖基于 SpringBoot 与 Vue 实现高校实验室预约管理系统，强调前后端分离模式有助于降低视图开发与业务逻辑之间的耦合[14]。黄孝新等从功能分析角度指出，高校实验室管理系统需要围绕实验室排课、开放预约、用户权限和数据权限等实际业务进行设计[15]。

从现有研究看，实验室预约管理系统已经具备较丰富的技术路线和实践案例，但仍存在以下不足。第一，部分系统主要面向单校区或单实验中心，跨校区资源协同和数据范围隔离设计不足。第二，部分研究重视页面功能实现，但对预约冲突检测、审批状态流转、角色权限边界等核心业务规则论述不够深入。第三，智慧校园和智能助手相关研究虽不断发展，但与实验室预约业务的结合仍有进一步拓展空间。因此，本文围绕跨校区协作场景，设计并实现一个集资源管理、预约审批、统计分析和智能问答于一体的分布式实验室管理系统。

### 1.3 研究内容

本文围绕“支持跨校区协作的分布式实验室管理系统”开展设计与实现，主要研究内容如下。

1. 多角色权限体系设计。结合系统使用场景，划分学生、教师、实验室管理员和系统管理员四类角色，明确不同角色在资源查看、预约提交、审批管理、数据维护和统计分析中的权限边界。
2. 跨校区资源模型设计。围绕校区、实验室、设备、用户、预约、审批和操作日志等对象建立数据模型，实现实验室资源的统一组织和跨校区管理。
3. 预约与审批业务实现。实现预约时间合法性校验、开放时间校验、容量校验、预约冲突检测、预约取消、审批通过与拒绝等核心业务逻辑。
4. 多端协同访问实现。面向管理员和普通用户分别提供 Web/H5 端与微信小程序端，通过统一后端 API 实现多终端业务复用。
5. 统计分析与智能助手实现。提供总览统计、校区维度统计和实验室利用率统计，并设计智能助手模块支持自然语言查询、预约引导和页面跳转。

### 1.4 技术路线

系统采用前后端分离与分层架构设计。后端基于 Flask 框架提供 RESTful API 服务，使用 SQLAlchemy 完成对象关系映射，支持 SQLite 开发环境和 MySQL 生产环境。认证授权采用 JWT 机制，结合基于角色的访问控制思想实现登录鉴权、角色权限校验和校区级数据范围控制。JWT 标准提供了一种紧凑的声明传递方式，适合前后端分离系统中的无状态认证场景[12]；RBAC 模型则为角色、权限和用户之间的映射提供了经典理论基础[13]。在接口设计方面，系统遵循 REST 架构风格，将业务资源抽象为接口路径，并通过统一响应结构降低前端处理复杂度。

前端部分采用 uni-app 实现 Web/H5 管理界面，微信小程序端采用原生小程序技术实现移动访问。两端均通过统一 API 调用后端服务，保证业务规则集中在后端 Service 层，避免多端重复实现造成逻辑不一致。智能助手模块采用规则模式作为基础能力，并预留 LLM 服务接入配置，用于支持实验室查询、排期查询、预约辅助和常见问题回答。

### 5.1 Agent 模块设计目标

可写内容：

- 降低用户查找实验室、查看排期和理解预约流程的操作成本。
- 将实验室查询、预约记录查询、统计概况查询等常用操作封装为自然语言入口。
- 为 Web 管理端和微信小程序端提供统一的智能问答接口，避免多端重复开发。
- 在规则模式可独立运行的基础上，预留 LLM 接入能力，提高系统后续扩展空间。

建议表述重点：Agent 不是泛聊天机器人，而是“面向实验室预约场景的智能辅助模块”。这样写更稳，也更符合毕业设计系统边界。

可引用：[10][11] 说明智慧校园背景，[12][13] 说明系统仍受认证与权限约束，[20][21] 说明 Agent 的“推理 + 行动/工具调用”设计思路。

### 5.2 Agent 总体架构设计

建议画图：用户端/管理端 -> Agent API -> Agent Service -> 意图识别层 -> 工具调用层 -> 业务服务/数据库 -> 结构化回复。

可写内容：

- 接口层：由 `/api/agent/chat` 接收用户输入、会话标识和当前登录用户信息。
- 服务层：`agent_service.py` 负责意图识别、参数抽取、工具选择和回复生成。
- 工具层：封装“我的预约”“可预约实验室”“实验室排期”“统计概况”等业务能力。
- 数据层：复用现有预约、实验室、校区、统计等模型与服务，不额外绕过业务规则。
- 输出层：返回文本回复、结构化数据和页面跳转动作，便于前端渲染和导航。

这一节要强调：Agent 并没有独立维护一套数据，而是复用系统已有 API、Service 和权限逻辑，从而保证回答结果与业务系统一致。

### 5.3 意图识别与规则模式设计

可写内容：

- 将用户输入划分为预约查询、实验室查询、排期查询、统计查询、帮助引导等意图。
- 通过关键词、日期表达式、实验室名称、校区名称等信息进行轻量参数抽取。
- 对无法识别的问题返回引导式回答，提示用户可以询问“我的预约”“明天有哪些实验室可预约”“某实验室排期”等。
- 规则模式不依赖外部大模型服务，保证系统在离线或未配置 LLM Key 的情况下仍可演示和运行。

可举例：

```text
用户：明天有哪些实验室可以预约？
系统识别：可预约实验室查询
抽取参数：日期=明天
调用工具：查询开放实验室与已有预约冲突
返回结果：可预约实验室列表 + 跳转预约页动作
```

### 5.4 工具调用与业务数据访问

建议结合项目源码写：

- 我的预约工具：根据当前用户 ID 查询个人预约记录，学生和教师只能查看自己的预约。
- 可预约实验室工具：结合实验室状态、开放时间、容量和已有预约记录筛选可用资源。
- 实验室排期工具：按实验室和日期查询 pending/approved 状态预约，辅助用户避开冲突时段。
- 统计概况工具：管理员可查看校区、实验室和预约数量等统计信息，普通用户不返回管理统计。
- 页面动作工具：返回前端可识别的跳转路径，例如实验室列表页、预约页、我的预约页。

这里要特别强调权限边界：Agent 的回答不能绕过 JWT 和 RBAC 控制，普通用户不能通过问答获得管理员统计或其他用户预约详情。可引用 JWT 和 RBAC 文献[12][13]。

### 5.5 LLM 可扩展模式设计

可写内容：

- 系统通过环境变量配置 `AGENT_PROVIDER`、`AGENT_MODEL`、`LLM_API_KEY` 和 `LLM_BASE_URL`。
- 默认使用规则模式，配置外部模型后可将自然语言理解和回复润色交给 LLM。
- LLM 只负责理解和生成表达，涉及数据库查询、预约状态、权限判断的内容仍由后端工具完成。
- 对 LLM 输出结果进行业务边界约束，避免生成与系统真实数据不一致的回答。

建议表述：本文实现的是“规则驱动为主、LLM 扩展为辅”的 Agent 架构，既保证演示稳定性，又保留后续智能化升级空间。LLM 扩展部分可结合 ReAct 和大语言模型智能体综述类文献进行说明[20][21]。

### 5.6 Agent 前端交互与应用效果

可写内容：

- Web/H5 端通过对话窗口展示用户问题、系统回答和快捷操作。
- 微信小程序端可复用同一接口，用于移动端快速查询预约与实验室信息。
- Agent 返回的 `actions` 可驱动页面跳转，减少用户从多个菜单中查找功能的成本。
- 在应用效果上，Agent 提升了系统的人机交互体验，使传统管理系统从“用户找功能”转向“用户表达需求，系统辅助定位功能”。

建议截图：

- Agent 首页/对话页。
- 查询“我的预约”的回复结果。
- 查询“可预约实验室”的回复结果。
- 返回页面跳转动作后的目标页面。

### 5.7 本章小结

本章总结 Agent 模块的创新性：它不是简单增加聊天窗口，而是将自然语言交互、业务工具调用、权限控制和多端页面动作结合到实验室预约业务中。该模块在不破坏原有业务架构的前提下，为用户提供更低门槛的资源查询和预约辅助能力，是本文区别于传统实验室管理系统的重要创新点。

## 第8章 总结与展望

### 8.1 工作总结

总结系统完成的模块：认证、资源管理、预约审批、统计分析、小程序端、Web 端、Agent。

### 8.2 创新点与不足

可写创新点：

- 面向跨校区协作场景设计资源与权限模型。
- 将预约冲突检测、审批流和数据范围控制集中在后端业务层。
- 同时提供 Web 管理端和微信小程序端。
- 单独设计面向实验室预约场景的 Agent 智能助手模块，将自然语言交互、业务工具调用、权限控制和页面跳转结合起来。

可写不足：

- 智能助手主要面向规则和轻量工具调用，复杂自然语言理解仍可增强。
- 系统尚未接入学校统一身份认证。
- 统计分析维度仍可扩展。
- 尚未进行大规模真实用户压力测试。

### 8.3 后续优化方向

- 接入统一身份认证和校园门户。
- 增加消息通知与审批提醒。
- 引入更精细化的排班和资源优化算法。
- 增强可视化大屏和数据分析能力。
- 扩展 Agent 的多轮预约、智能推荐和异常解释能力。
